Find Discounts

Goal:

The goal of this use case is to locate any discounts associated with a shopping list. This action will determine if any coupons or sales exist for each item on the shopping list. If a discount is found for an item it will be indicated  in the shopping list UI for that item.

 

Description: This table list the steps of the scenario from trigger to successful goal attainment.

 

Step

Action

1  User selects Find Discounts option from Shopping List UI Menu
2 The system determines if it knows which Grocery Store Finder to use <has location service detected user is within range of a grocery store> If store can't be determined then go to step 2a.
3 The system looks up the correct Grocery Store Finder to use for determining sales discounts and coupons for each item in the shopping list. A Discount entity is created for each item where a possible discount is available.
4 The system places an indicator next to each shopping list item where a possible Discount is available.

5

 

 

Variations: Branching Actions

 

       

Step

Action

2a  System displays grocery store list and request user selection, if no stores in list then handle error: GROCERY_STORE_LIST_EMPTY
2b  The user selects a Grocery Store from the list

Related Information:

 

Priority  This Use case is highly ranked because it represents a fundamental capability of the Application. This Use Case will be a major capability that sets the product apart from others in the market place.
Performance

Though no exact limit is has yet been determined it is important that this Use case execute in a timely manner. The user should not expect this process to take too long such that it is apparent and adversely affects the shopping experience.

 

For this reason this "Finder" processing should take place in a separate thread of execution so that the user can still interact with the system and do other tasks. As the discounts process is completed for items their UI's are adjusted to show this fact.

Frequency/Scale

This process should be a fairly infrequent event. This should only occur when a shopping trip is starting or being planned. For most people this happens between once a week to once every other week.

 

Since the finders are store specific and shopping lists are active for one particular store for a shopping event there should only be one finder running at a time. If the user wishes to purchase some items on the list at another store they are simply not purchased and when the user is in another store the Discounts will be recreated for that store.

 

The Discount finder will have to process all the items on a shopping list. Typically this should be around 20 to 30 items. However there will be shopping events that may contain many more items.

Reliability / Availability

The GRUB system should not be affected by failures in the Discount Finder subsystem. If the Discount Finder subsystem fails the information is presented to the user but the shopping list and the shopping process should not be affected, other that no discount information will be available.

Security None
Due Date At the very least the infrastructure needed to implement this use case should be completed by 9/30/2009. Also at least one Grocery Store Finder should be implemented even if that means it is purely mocked up.
Open Issues  The major issue yet to be resolved with this use case is tacking the problem of how to find coupon and sore discounts to specific grocery stores. Most stores present this data on their Web sites but it is usually in a form that is nearly impossible to obtain via automated processes. And the legality of copying this data from a web page may be an issue.

GRUB

Grocery Repository and Utilities Bundle (GRUB). This system is a grocery management application that provides services to identify items to purchase and adds them to shopping lists. Shopping lists provide utilities to aid with the actual shopping process. Once items are purchased they are stored in a pantry which is used to track inventory. When items are identified as being depleted they are automatically added to a shopping list.

 

In addition to the above tools the system will provide mechanisms to determine if items in a shopping list have any discount associated with them (sale or coupon). Also the pantry information is used to match recipes against. Users can search for recipes that contain various items available in the pantry or can determine if the ingredients identified in a recipe are available in their current pantry inventory.

Requirement Diagram
Pantry

Summary

Name Documentation
Pantry The GRUB Application shall contain a Pantry component. The pantry is an abstraction of a home pantry. This is the repository where groceries are stored and inventoried. Once Items are marked as purchased from a shopping list they are added to the pantry.
Pantry Items will be displayed in a Linear List

The Pantry shall display the pantry items items as a list.

Pantry Item Meta-Data Pantry items shall be describe via a combination of the Catalog meat-data and the additional pantry meta-data listed: 
  • Quantity or Size.
  • Last purchased location (store or market name)
  • Last price paid
  • Staple (a flag indicating the item is required to be in repository )
  • Out (a flag indicating the item is depleted)
Pantry Searching

Users have the ability to browse or search the Pantry

    • Using various queries, including default queries and user defined queries.
    • Using voice recognition look up by item name
Pantry Bootstrap Mode

A special mode will exists where catalog items can be added directly to the pantry without being purchased from a shopping list. Users shall select this option from the Pantry Menu - BootStrap mode. This brings up a special catalog view where and items selected are added directly to the pantry.

Adding Items to Shopping List

Users shall have the ability to add items form the Pantry to the Default Shopping List or any other Shopping Lists at any time.

Pantry Advanced Item Search

Users shall have the ability to search for items using advanced search criteria. This includes searching by:

  • Category
  • Food Type
  • Inventory Status (out or not)
  • Staple Flag
Manual shopping List Insertion

Users shall have the ability to manually add items to a shopping list.

Automatic Shopping list Removal on Reset If the inventory state of an item in the pantry is reset from out then the item shall be removed from the shopping list.
Automatic Shopping list Insertion Pantry items marked as Out are automatically added to the default shopping list.
Pantry Voice Search Users shall have the ability to search the Pantry for specific items using the items unique name.
Predictive Search

Users shall have the ability to search the pantry using predictive search.


Select Multiple Items and Assign to a Specific Shopping List

Users shall have the ability to select multiple items in the Pantry and add to Shopping List. Users shall select multiple Items and then choose the Pantry Menu - Add To Shopping List. This then presents the user with a submenu listing all the available shopping list options.

Short Touch to Add Item to Default Shopping List Users shall have the ability to add a Pantry item to the Default list by simply doing a short touch on the item in the Pantry list view.
Long touch to Add Item to Specific Shopping List

Users shall have add items to specific shopping list by long touching a Pantry Item. This shall bring up them Items Context Menu and present the user with the Pantry Item Context Menu. one of the items on the Context menu will be an option to add to a shopping list choice Menu.
